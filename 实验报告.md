## 一、实验目的

参考"实验七 使用 Hardhat 框架搭建宠物商店 Dapp"的技术路线，独立设计并实现一个基于以太坊智能合约的去中心化众筹平台，打通"**合约编写 → 单元测试 → 本地链部署 → Web 前端交互**"的完整开发链路。在完成核心众筹功能的基础上，重点强化**链上（智能合约）部分的深度**，实现：

1. 简化版**里程碑分期资金释放**（基于目标完成度 + 发起人标记）；
2. 基于到达顺序的**早鸟奖励**机制；
3. **捐赠者治理（DAO）**——里程碑放款由出资人按权重链上投票决定；
4. **奖励代币（ERC-20）与早鸟徽章（ERC-721）**——捐赠自动发放，展示代币标准与合约间调用；
5. **重入攻击演示与防御**——呼应"实验八 智能合约攻击"，用攻击合约验证安全设计；
6. **开发工具链**——自定义 Hardhat 命令行任务与 Gas 用量报告。

同时重点解决里程碑功能与"项目失败后全额退款"规则之间的逻辑冲突。

## 四、系统设计

### 4.1 合约架构（多合约协作）

```
        部署接线(setMinter)
   ┌─────────────────────────────┐
   ▼                             │
BackerToken (ERC-20)        EarlyBirdBadge (ERC-721)
   ▲  mint(捐赠奖励)              ▲  mint(早鸟徽章)
   │                             │
   └──────── Crowdfunding ───────┘
             (主合约：项目/捐赠/里程碑/治理/退款)
                   ▲
            ReentrancyAttacker（安全测试用，不部署到生产）
```

`Crowdfunding` 在构造时注入 `BackerToken` 与 `EarlyBirdBadge` 地址；部署脚本随后把两者的铸造权（minter）授予众筹合约，使捐赠时可自动发放奖励代币与徽章。前端按当前 chainId 自动读取 Ignition 生成的部署地址与 ABI。

### 4.2 数据结构

`Project` 结构体核心字段：`id / creator / name / description / goal / deadline / totalRaised / released（已释放额）/ state / donorCount / milestoneCount / milestonesReleased / requireVote（是否启用治理投票）`。

辅助映射：`contributions`（累计捐赠，退款后清零）、`earlyBirdRank`（1–10 早鸟排名）、`milestones`（里程碑比例）、`proposals`（当前里程碑放款提案）、`hasVoted`（投票去重）。

### 4.3 状态机

```
create ─▶ Active ──finalize(到期)──┬─ raised≥goal ─▶ Successful ─ withdraw / releaseMilestone
   │ donate / 里程碑可分期释放        └─ raised<goal ─▶ Failed ─ refund(按未释放比例)
```

里程碑释放允许在 Active 或 Successful 触发；Failed 后禁止释放，保证退款资金不被抽走。

## 五、核心合约逻辑

### 5.1 创建 / 捐赠 / 结束

`createProject` 校验参数与里程碑合计 100，分配自增 ID。`donate` 要求 Active 且未过期，累加贡献与总额，按到达顺序登记前 10 名早鸟，并**自动发放奖励**（见 5.4）。`finalize` 到期后按是否达标置为 Successful/Failed，任何人可调用。

### 5.2 里程碑分期释放 `releaseMilestone`

第 `idx` 个里程碑解锁需"目标完成度达标"：`totalRaised × 100 ≥ goal × (前 idx+1 个比例之和)`；释放金额 `= goal × percent[idx] / 100`。普通项目"发起人调用即代表标记完成"；治理项目还需提案通过（见 5.3）。转账遵循 Checks-Effects-Interactions 并加重入锁。

### 5.3 捐赠者治理投票（DAO）⭐

启用 `requireVote` 的项目，里程碑放款须经捐赠者投票：

1. `proposeMilestone(id)`：发起人在下一里程碑达完成度后发起放款提案；
2. `voteOnMilestone(id, support)`：捐赠者按**出资额为权重**投票，每个里程碑每人一票；
3. 通过条件：`yesWeight × 2 > totalRaised`（赞成权重超过总筹集额一半）；
4. `releaseMilestone(id)`：仅当提案通过时才放款，随后关闭提案。

这把"是否放款"的决定权从发起人单方转移到全体出资人，体现区块链**去中心化治理**思想。权重按出资额而非人头，避免女巫攻击（一人多地址刷票无收益）。

### 5.4 奖励代币与早鸟徽章（ERC-20 / ERC-721）⭐

- **BackerToken（BACK，ERC-20）**：自实现最小 ERC-20（`balanceOf/transfer/approve/transferFrom/事件`）。捐赠时按 `1 BACK / ETH` 铸造，早鸟额外 +20%。
- **EarlyBirdBadge（EBADGE，ERC-721）**：自实现最小 ERC-721（`ownerOf/balanceOf/transferFrom/tokenURI/supportsInterface`）。每位早鸟首次捐赠时铸造一枚链上唯一徽章，`tokenURI` 形如 `ChainFund Early Bird #3 @project0`。

二者均设 `minter` 权限，仅众筹合约可铸造；徽章用 `_mint`（非 `safeMint`）避免回调，杜绝铸造引入的重入面。该设计展示了对**代币标准**与**合约间调用（cross-contract call）**的掌握。

### 5.5 发起人提款与失败退款

`withdraw`：成功后发起人提取剩余资金（无里程碑→全部；有里程碑→所有里程碑释放后的超额）。`refund`：失败后捐赠者按未释放比例退款（见第六节）。

## 六、里程碑与"全额退款"的冲突分析与解决 ⭐

**冲突来源**：传统众筹失败时合约托管着全部资金，故可"全额退款"。但里程碑允许发起人在项目结束前分批提走部分资金；一旦项目失败，合约余额已少于 `totalRaised`，无法让每人全额退款。

**解决方案**：退款额改为按未释放比例：

$$\text{可退}_i = \text{捐赠}_i \times \frac{\text{totalRaised} - \text{released}}{\text{totalRaised}}$$

**守恒性**：失败后 `totalRaised` 与 `released` 冻结，所有人可退之和

$$\sum_i \text{捐赠}_i \cdot \frac{U}{T} = \frac{U}{T}\cdot T = U = \text{totalRaised}-\text{released}$$

恰好等于合约剩余余额，不多不少。**向后兼容**：无里程碑项目 `released≡0`，公式退化为全额退款，与原规则一致。

## 七、安全性：重入攻击演示与防御 ⭐

呼应"实验八 智能合约攻击"，编写 `ReentrancyAttacker`：在收到退款的 `receive` 回调中再次调用 `refund`，企图重复取款掏空合约。

**防御措施**：`refund/withdraw/releaseMilestone` 均（1）采用 **Checks-Effects-Interactions**——先把 `contributions` 清零 / 更新 `released`，再 `call` 转账；（2）加 `noReentrant` 互斥锁。攻击者再入时，要么命中重入锁 `require(!locked)` 回滚，要么因余额已清零而无利可图。

**测试验证**：攻击者捐 2 ETH 的失败项目，发起攻击后其 `reentryCount > 0`（确实尝试重入）但余额恰为 2 ETH（未多拿），合约余额清零——证明防御有效。

## 八、前端实现

单页仪表盘：顶部钱包栏显示**地址 / BACK 代币余额 / 早鸟徽章数量**，并支持**深色模式**切换。主体含统计概览、创建表单（可勾选"启用治理投票"）、项目卡片、右侧**链上动态流**。

## 十一、关键技术点小结

- **去中心化治理**：出资权重投票决定里程碑放款，削弱单点信任。
- **代币标准**：从零实现 ERC-20 与 ERC-721，理解 mint / 事件 / 接口探测（ERC-165）。
- **合约间调用**：主合约通过接口调用代币/徽章合约铸造，配合 minter 权限控制。
- **智能合约安全**：CEI + 重入锁，并用攻击合约实测防御（呼应 lab8）。
- **事件驱动 DApp**：前端用 `queryFilter`/`on` 订阅链上事件实现实时动态，而非轮询。
- **工程化**：Ignition 多合约有序部署与接线、自定义 CLI 任务、Gas 成本量化。

## 十二、总结与改进方向

本项目在完成众筹核心功能的基础上，重点深化了链上部分：DAO 治理投票、ERC-20/ERC-721 双代币、重入攻击防御、事件驱动前端与开发工具链，并以"未释放部分按比例退款"严谨地解决了里程碑与全额退款的冲突，17 个单元测试全部通过。后续可进一步：里程碑提案加入投票截止时间与法定人数（quorum）、早鸟徽章接入链下元数据与图片、支持 ERC-20 代币捐赠、以及用 The Graph 建立事件索引以优化大规模查询。
